# 아키텍처 문서 작성 지침

소프트웨어 아키텍처(High-Level Design)는 시스템의 '큰 그림'을 그리는 작업이다.
무엇을 포함하고 배제할지의 기준은 **"이 결정이 시스템의 주요 특성(품질 속성)과 구조에 영향을 미치는가?"** 에 달려 있다.

---

## 1. 아키텍처에 담아야 할 것 (What to Include)

아키텍처 설계서는 시스템의 뼈대와 핵심 원칙을 정의해야 한다.

- **시스템의 구조적 분할:** 컴포넌트, 모듈, 레이어 간의 관계와 역할 분담.
- **핵심 품질 속성 (Quality Attributes):** 성능, 가용성, 확장성, 보안성 등을 어떻게 달성할 것인가에 대한 전략.
- **컴포넌트 간 통신 방식:** REST API, gRPC, 메시지 큐(Kafka/RabbitMQ) 등 데이터가 흐르는 방식.
- **데이터 저장 전략:** RDBMS를 쓸 것인지, NoSQL을 쓸 것인지, 혹은 분산 파일 시스템을 쓸 것인지에 대한 유형 정의.
- **기술 스택의 제약 사항:** 조직의 표준이나 인프라 환경에 따른 필수적인 기술적 선택.

---

## 2. 아키텍처에 담지 말아야 할 것 (What to Exclude)

상세 설계나 구현 단계에서 결정해도 되는 세부 사항은 과감히 배제한다.

- **구현 알고리즘:** 특정 기능을 구현하는 내부 로직이나 복잡한 소스 코드 수준의 디테일.
- **데이터베이스 스키마의 세부 필드:** 테이블의 컬럼명, 데이터 타입 등은 상세 설계의 영역이다.
- **UI/UX의 미세한 요소:** 버튼의 위치, 색상, 폰트 등 시각적 디테일.
- **자명한 유틸리티 라이브러리:** 날짜 계산을 위해 어떤 라이브러리를 쓸지와 같은 사소한 선택.

---

## 3. 특정 기술명 포함 기준: "Postgres를 쓴다"는 담아야 할까?

정답은 **"어느 수준까지 결정되었느냐"** 에 달려 있다.

### 포함해야 하는 경우 (의사결정의 근거가 있을 때)

단순히 이름만 적는 것이 아니라, "왜 Postgres인가?"에 대한 아키텍처적 이유가 있다면 포함해야 한다.

> "우리 시스템은 복잡한 관계형 쿼리와 데이터 무결성이 핵심이므로, ACID를 보장하며 확장성이 검증된 **RDBMS(Postgres)** 를 사용한다."

### 배제해도 되는 경우 (유연성이 필요할 때)

상위 설계 단계에서 특정 제품명에 매몰되면 나중에 기술을 변경하기 어려워진다. 이럴 때는 **추상화된** 표현을 쓴다.

> "데이터의 일관성을 위해 **표준 SQL을 지원하는 관계형 데이터베이스(RDBMS)** 를 사용한다."

## 작업 방식

### 아키텍처 문서 작성 요청 시

1. PRD(기획서)를 먼저 읽어 시스템의 목적과 비기능 요구사항을 파악한다.
2. 위 기준에 따라 포함할 내용과 배제할 내용을 분류한다.
3. 각 기술 선택에 대해 **"왜 이 기술인가?"** 의 아키텍처적 근거를 명시한다.
4. 품질 속성(성능, 보안, 확장성 등)별로 달성 전략을 구체적으로 서술한다.
5. v1.0 범위에서 의도적으로 제외한 아키텍처 요소도 명시하고 제외 이유를 기록한다.

### 아키텍처 문서 리뷰 요청 시

다음 항목을 점검하고 문제가 있으면 구체적으로 지적한다.

- [ ] 구조적 분할이 명확한가? (레이어, 컴포넌트 역할이 정의되어 있는가)
- [ ] 품질 속성별 달성 전략이 있는가?
- [ ] 통신 방식이 명시되어 있는가?
- [ ] 데이터 저장 전략에 근거가 있는가? (추상화 vs 구체화 판단 포함)
- [ ] 기술 선택에 "왜?"가 붙어 있는가?
- [ ] DB 스키마 세부 필드, UI 디테일, 알고리즘이 혼입되어 있지 않은가?
- [ ] 범위 외 결정이 명시되어 있는가?
