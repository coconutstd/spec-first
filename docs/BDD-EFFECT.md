# BDD가 기획 구멍을 찾아낸 사례

**프로젝트:** 단순 게시판 서비스  
**작성일:** 2026-04-29  
**목적:** BDD 시나리오 작성이 기획 결함 발견에 얼마나 효과적인가를 기록한 케이스 스터디

---

## 배경

PRD, Architecture, ERD, BDD 문서를 순서대로 작성한 뒤, BDD 시나리오의 관점에서 PRD를 역으로 검토했다. BDD는 "구체적인 입력 → 구체적인 출력"을 강제하기 때문에 추상적인 기획 문장이 감추고 있던 결함이 드러났다.

## 결정된 해결책

| # | 결정 내용 |
|---|---|
| 1 | 응답 body에 구분자 설계: `{ "code": "POST_DELETED" }` vs `{ "code": "POST_NOT_FOUND" }` |
| 2 | `/posts/:id/edit` 직접 접근 차단 (서버 사이드에서 비밀번호 검증 토큰 없으면 리다이렉트) |

## BDD가 효과적이었던 이유

### 1. 구체적 시나리오가 추상적 문장의 빈틈을 드러낸다

PRD에 "삭제된 게시글에 접근하면 안내 메시지가 표시된다"고 썼을 때는 문제가 없어 보였다. BDD 시나리오를 작성할 때 백엔드 Then을 "404를 반환한다"로 쓰고, 프론트엔드 Then을 "삭제된 게시글입니다 표시"로 썼더니 — **같은 404 응답으로 두 가지 다른 메시지를 어떻게 구분하나**라는 질문이 자동으로 생겼다.

### 2. Given-When-Then이 경계값을 강제한다

PRD "닉네임: 2~20자"는 하한(1자 에러)과 상한(21자 에러) 두 개의 시나리오를 요구한다. BDD를 쓰다 보면 "상한 시나리오도 써야 하는데 PRD에 상한 처리가 없다"는 것을 자연스럽게 발견한다.

### 3. Feature 간 상호작용이 정책 결정을 요구한다

"게시글 삭제" Feature와 "댓글 조회" Feature를 따로 쓰다가, **삭제된 게시글의 댓글 URL로 직접 접근하면?** 이라는 시나리오를 작성해야 하는 순간 댓글 cascade 정책이 없다는 게 드러났다.

### 4. 보안 시나리오가 플로우 외 접근을 강제한다

수정 플로우를 시나리오로 쓰다 보면 자연스럽게 "정상 플로우가 아닌 경우"를 작성하게 된다. URL 직접 접근 시나리오를 쓰려는 순간 PRD에 그에 대한 정책이 없다는 걸 발견했다.

---

## 결론

BDD 시나리오 작성은 단순한 테스트 문서가 아니다. **기획서를 실행 가능한 형태로 번역하는 과정**에서 기획의 모호함, 경계값 미정의, 정책 누락을 조기에 발견하는 설계 검증 도구다. 이 프로젝트에서 코드 한 줄 작성 전에 10개의 기획 구멍을 발견했고, 구현 전에 모두 결정했다.
